# Ревью проекта "RAG Catalog" (Semantic Search)

В ходе анализа структуры и кодовой базы проекта (на базе Qdrant, sentence-transformers, NiceGUI, PyQt6) были выявлены следующие сильные стороны и области для улучшения.

## 1. Архитектура и структура
> [!NOTE]
> Архитектура грамотно разделена на ядро (`src/rag_catalog/core`), пользовательские интерфейсы (`src/rag_catalog/ui`), интеграции (`telegram_bot.py`) и инструменты командной строки (`src/rag_catalog/cli`).

**Плюсы:**
* **Модульность точки входа**: Файлы в корне проекта (`index_rag.py`, `rag_search.py`, `nice_app.py`) служат легкими обертками (shims), которые импортируют реальную логику из `src/rag_catalog`. Это предотвращает проблемы с относительными импортами.
* **Поддержка форматов и OCR**: Отличная реализация fallback-механизмов (pymupdf -> pdfplumber -> OCR через pytesseract/pdf2image) в зависимости от наличия текстового слоя в PDF.
* **Инкрементальное индексирование**: Логика разделения индексации на стадии (`metadata`, `small`, `large`) — это отличный подход для предоставления быстрого доступа к поиску на больших базах.

## 2. Качество кода и масштабируемость
> [!WARNING]
> Некоторые ключевые файлы стали слишком монолитными (God Objects), что усложняет их поддержку и тестирование.

**Проблемные места:**
* `src/rag_catalog/core/index_rag.py` (более 1500 строк). В нем смешаны логика работы с Qdrant, управление состоянием (state.json), обход файлов и все парсеры форматов (DOCX, XLSX, PDF, Images).
  **Рекомендация:** Вынести экстракторы текста (`_extract_docx`, `_extract_pdf` и т.д.) в отдельный модуль `extractors.py`.
* `src/rag_catalog/ui/nice_app.py` (более 3700 строк). Файл содержит всю UI логику, работу с БД для телеметрии и аутентификации, а также управление состоянием `PageState`.
  **Рекомендация:** Разделить на логические компоненты (например, вынести настройки профиля, компонент поиска, компонент просмотра файлов в подмодули).

## 3. Выявленные ошибки (Тестирование)
> [!IMPORTANT]
> При запуске тестов (`pytest -q tests`) обнаружено **3 упавших теста** в модуле `tests/test_indexer_stage_logic.py`. 

1. `test_chunk_text_has_overlap`: Метод `_chunk_text` возвращает `["abcde", "defgh", "ghij", "j"]` вместо ожидаемых трех элементов, так как последний чанк "хватает" оставшийся один символ `'j'`. Стоит либо поправить тест, либо обновить логику чанкинга (например, игнорировать "хвосты" короче определенного размера).
2. `test_small_stage_file_without_content_stays_metadata` 
3. `test_small_stage_file_with_content_becomes_content`
Упавшие тесты стадийности говорят о том, что рефакторинг инкрементального индекса сломал логику `_should_skip_for_stage` (или в тестах используются устаревшие моки). 

## 4. Рекомендации по улучшению

1. **Исправить упавшие тесты**, чтобы гарантировать надежность инкрементального индексатора.
2. **Внедрить линтеры/форматтеры**: Добавить `ruff` или `flake8` и `black` в `pyproject.toml` (раздел `dev`). Сейчас есть только скрипт `check_syntax.py`, который выполняет базовую проверку `ast.parse`.
3. **Рефакторинг монолитов**: Начать с декомпозиции `nice_app.py` на страницы/роуты (например, `/search`, `/settings`, `/explorer`), используя встроенный в NiceGUI роутинг (`@ui.page`).
4. **Контроль ресурсов**: Убедиться, что `ThreadPoolExecutor` при чтении/индексировании не исчерпывает оперативную память (сейчас используется пакетная загрузка, что хорошо, но стоит дополнительно отслеживать RAM для `_extract_pdf` с OCR).

---
**Итог:** Проект выглядит мощным, выполняет важную бизнес-задачу и имеет хороший фундамент (Qdrant + локальные эмбеддинги). Основная точка роста — борьба с техническим долгом в виде больших файлов и актуализация тестов.
