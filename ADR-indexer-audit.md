# ADR-001: Аудит архитектуры индексатора RAG

**Статус:** В работе  
**Дата:** 2026-05-21  
**Область:** `src/rag_catalog/core/index_rag.py`, `indexing/stage_runner.py`, `index_state_db.py`, `chunking/text.py`, `extractors/files.py`

---

## Контекст

Индексатор прошёл путь от монолитного скрипта (один проход, JSON state) до
многоэтапного конвейера с SQLite, batch-encode и pipeline I/O. Код рабочий и
хорошо документирован внутри, но накопил ряд проблем — от архитектурных
странностей до серьёзных пробелов в функциональности.

---

## 1. Иррациональные и абсурдные функции

### 1.1 Зомби-методы: `_index_metadata` / `_index_content` / `_extract_xlsx` / `_extract_xls`

**Проблема.** `RAGIndexer` содержит методы `_index_metadata` и `_index_content`,
которые кодируют векторы *по одному файлу* вызовом `encode()`. Весь современный
pipeline (через `IndexStageRunner.run()`) обходит их стороной и делает batch-encode
напрямую. Методы вызываются только из `process_file()` — legacy-пути для
разовой обработки одного файла.

Аналогично, `_extract_xlsx` и `_extract_xls` дублируют `_extract_spreadsheet`:
в pipeline используется только последний, первые два — мёртвый код внутри класса.

**Вред:** читатель кода тратит время на понимание трёх путей, где реально работает
один. Тесты, которые мокают `_extract_xlsx`, тестируют несуществующий маршрут.

**Рекомендация:** удалить `_index_metadata`, `_index_content`, `_extract_xlsx`,
`_extract_xls` из `RAGIndexer`. `process_file()` переписать через
`IndexStageRunner` или сделать тонкой обёрткой над ним.

---

### 1.2 `_create_collection` — метод-призрак

```python
def _create_collection(self) -> None:
    ensure_collection(..., recreate=False)
```

Это в точности то, что делает `_setup_collection(recreate=False)`. Метод нигде
не вызывается, существует только как наследие рефакторинга.

**Рекомендация:** удалить.

---

### 1.3 `_save_state` — метод-пустышка

```python
def _save_state(self) -> None:
    """Compatibility shim for legacy JSON state (no-op with SQLite)."""
    return
```

Вызывается в пяти местах кода. Каждый вызов — это шум при чтении: «что здесь
сохраняется?». Фактически — ничего.

**Рекомендация:** удалить вызовы, удалить метод, добавить комментарий в git-историю.

---

### 1.4 `state: Dict = {"files": {}}` — in-memory зеркало SQLite

```python
self.state: Dict[str, Any] = {"files": {}}
```

Создаётся в `__init__` «для тестов, которые ожидают `idx.state["files"][path]`».
Это означает, что часть тестов тестирует внутреннее состояние устаревшего API,
а не поведение. При каждом `_upsert_state_entry` поддерживается синхронизация с
SQLite — потенциальный рассинхрон при больших объёмах.

**Рекомендация:** исправить тесты, убрать зеркало. Если тесты проверяют
persistence — пусть читают из `state_db.get_entry()`.

---

### 1.5 `IndexStageRunner` как прокси через `__getattr__`/`__setattr__`

```python
class IndexStageRunner:
    def __getattr__(self, name): return getattr(self._indexer, name)
    def __setattr__(self, name, value):
        if name.startswith("_"): object.__setattr__(self, name, value)
        else: setattr(self._indexer, name, value)
```

`IndexStageRunner` оборачивает `RAGIndexer` через магические методы и
прозрачно делегирует все обращения к атрибутам. Это означает, что
`self.catalog_path` внутри `run()` — это `self._indexer.catalog_path`.
Паттерн скрывает зависимости, ломает автодополнение IDE и делает дебаг
неочевидным. Весь `run()` при этом — один метод.

**Рекомендация:** сделать явную передачу параметров в конструктор или
вынести `run()` как свободную функцию, принимающую `indexer` первым аргументом.

---

### 1.6 Магическое число `4` в `parent_id` провенанса

```python
parent_id = f"{doc_id}:chunk-group:{chunk_index // 4}"
```

Чанки группируются в «группы» по 4 для формирования иерархии
`doc → group → chunk`. Число 4 захардкожено без объяснения. Размер группы
никак не связан с `chunk_size`, семантикой документа или chunk_overlap.

**Рекомендация:** либо удалить эту псевдоиерархию (достаточно `doc_id` +
`chunk_index`), либо сделать `chunk_group_size` конфигурируемым с объяснением
зачем.

---

### 1.7 Структура чанков как текстовые маркеры вместо метаданных

Провенанс (страница, лист, строка) хранится в *тексте чанка* в виде маркеров:

```
Страница: 5
Лист: Смета
Строка: 123
```

Затем три static-метода (`_extract_marker_int`, `_extract_marker_text`,
`_extract_section_title`) парсят эти маркеры обратно из текста регулярками.
Это означает: текст чанка загрязнён служебными строками, регулярки могут
конфликтовать с реальным содержимым, провенанс нельзя использовать без
десериализации.

**Рекомендация:** хранить провенанс *только* в payload Qdrant, не встраивать
в текст чанка. Это ускорит embed-кодирование и исключит ложные совпадения.

---

### 1.8 `DEFAULT_SYNONYM_MAP` захардкожен в индексаторе

120+ строк отраслевых синонимов (CAT, Komatsu, ПСМ, ОСАГО…) внутри
`index_rag.py`. Они смешаны с кодом, их сложно редактировать не-разработчику,
не версионируются отдельно от кода.

**Рекомендация:** вынести в `data/synonym_map.json` или `config/synonyms.yaml`,
загружать при старте. config.json уже поддерживает `synonym_map` для дополнения —
сделать то же для базового словаря.

---

### 1.9 Бесконечный цикл ожидания каталога в `__init__`

```python
while not self.catalog_path.exists():
    _time.sleep(_wait)   # _wait = 60 секунд
```

Если каталог недоступен, `__init__` зависает навсегда в главном потоке.
Нет таймаута, нет счётчика попыток, нет возможности прервать без kill-9.

**Рекомендация:** заменить на конечное ожидание (например, 10 попыток × 60с)
с `raise RuntimeError` при исчерпании.

---

### 1.10 `MAX_IMAGE_PAGES = 50` — захардкожен, остальные пороги — в конфиге

Пороги `small_office_mb`, `small_pdf_mb`, `max_chunks_per_file`, `read_workers`
— все конфигурируемы через `config.json` и CLI. `MAX_IMAGE_PAGES` — нет.
Для каталогов с многостраничными TIFF (технические чертежи, сканы схем) 50
страниц может быть либо мало, либо избыточно.

**Рекомендация:** добавить `ocr_max_image_pages` в конфиг.

---

### 1.11 `file_stage` при отсутствии контента на этапе small/large

```python
file_stage = "metadata" if stage == "metadata" else (
    "content" if result.get("has_content") else stage
)
if stage in ("small", "large") and not result.get("has_content"):
    logger.warning("...сохраняю stage=%s (повторного прохода не будет)...")
```

Если файл `.docx` или `.pdf` вернул пустой текст на этапе `small`, он
сохраняется со `stage="small"`. Это означает, что он **никогда не попадёт**
на этап `large` (skip-логика проверяет `existing_stage in ("content", self.current_stage)`).
Повреждённый или нечитаемый файл «застревает» в индексе навсегда — только
с метаданными, без контента, без шанса на переиндексацию.

**Рекомендация:** ввести отдельный stage-статус `"error"` или `"empty"` для
таких файлов, и при следующем прогоне всё-таки пробовать их переиндексировать.

## 2. Жизненно необходимые функции, которых нет

### 2.1 Исключения папок/файлов из индексирования

Нет механизма `.ragignore` или `exclude_patterns` в конфиге. Временные папки
(`~$`, уже фильтруется), папки `node_modules`, архивы резервных копий, системные
папки `.git` — всё попадает в индекс или требует патча кода.

**Нужно:** `exclude_paths` / `exclude_patterns` (glob/regex) в `config.json`.

---

### 2.2 Поддержка `.txt`, `.csv`, `.rtf`, `.pptx`, `.doc`

Из офисных форматов поддерживаются только `.docx`, `.xlsx`, `.xls`, `.pdf`.
Недостающие:

| Формат | Распространённость | Сложность добавления |
|--------|-------------------|----------------------|
| `.txt` | высокая | тривиально |
| `.csv` | высокая | тривиально |
| `.rtf` | средняя | `striprtf` |
| `.pptx` | средняя | `python-pptx` |
| `.doc` | средняя | `antiword` / `LibreOffice` |

Для каталога строительной документации `.txt` и `.csv` особенно критичны
(спецификации, ведомости в plain-text).

---

### 2.3 Retry для файлов с ошибками

Если файл дал ошибку при чтении, он попадает в счётчик `error_files` и
пропускается. При следующем прогоне он *переиндексируется* только если
изменился fingerprint. Временно недоступные файлы (SMB-сбой, lock) навсегда
остаются неиндексированными до следующего изменения файла.

**Нужно:** отдельный список `failed_paths` в `IndexStateDB` с retry-счётчиком
и полем `next_retry_at`. Это особенно важно на сетевых дисках (SMB).

---

### 2.4 Watch-режим (инкрементальная индексация по событию)

Сейчас индексация только polling: полный `rglob()` → сравнение fingerprint.
При 58 000 файлах это минуты даже на этапе `metadata`. Никакой реакции на
«файл только что сохранён».

**Нужно:** `--watch` режим через `watchdog` (кросс-платформенный FSEvents /
inotify / ReadDirectoryChangesW). Изменённый файл попадает в очередь и
индексируется в течение секунд.

---

### 2.5 Приоритетная очередь файлов

`rglob()` возвращает файлы в порядке файловой системы. Новые/изменённые файлы
не имеют приоритета перед старыми неизменёнными. При прерывании прогона нет
гарантии, что самые свежие документы проиндексированы первыми.

**Нужно:** сортировка `scope_files` по `mtime DESC` — новые файлы обрабатываются
первыми. Тривиальное изменение с высоким практическим эффектом.

---

### 2.6 Дедупликация контента

Одинаковые файлы в разных папках (копии, бэкапы) индексируются независимо.
Поиск возвращает дубли с одинаковым текстом. Нет детекции на уровне хеша
содержимого.

**Нужно:** при финальной записи в Qdrant проверять `content_hash` (MD5/SHA256
первых N байт текста). Дубли — пропускать или индексировать с `is_duplicate=True`.

---

### 2.7 Детекция несовместимости embedding-модели

При смене `embedding_model` в конфиге старые векторы в Qdrant несовместимы
с новой моделью (разная размерность, разное семантическое пространство).
Индексатор тихо дописывает новые векторы рядом со старыми — коллекция
становится мусорной.

**Нужно:** хранить `embedding_model` + `vector_size` в метаданных коллекции
(или в отдельной таблице `index_config`). При несовпадении — предупреждать
и требовать явного `--recreate`.

---

### 2.8 Статистика качества индексирования

Телеметрия считает добавленные/обновлённые/пропущенные файлы, но не качество:

- какой % PDF-файлов имел пустой текстовый слой и потребовал OCR
- средний размер чанка (признак плохого разбиения)
- % файлов с `error` на каждом этапе
- распределение `stage` в `index_state_db`

Без этих данных невозможно понять, насколько хорошо проиндексирован каталог.

**Нужно:** метод `RAGIndexer.quality_report()` или расширение `TelemetryDB.stats()`.

---

### 2.9 Поддержка архивов `.zip`

Документация на технику нередко поставляется в zip-архивах. Сейчас они не
индексируются вообще (расширение не в `SUPPORTED_EXTENSIONS`).

**Нужно:** экстрактор, распаковывающий zip во временную папку и индексирующий
содержимое с `path` вида `archive.zip/doc.pdf`.

---

### 2.10 Явный `--dry-run`

Нет способа узнать, *что* будет переиндексировано, не запуская индексацию.
`--dry-run` выводил бы список файлов к обработке с указанием причины
(новый / изменился / апгрейд stage).

### Реализация 2026-05-21

- `empty` stage сохраняется для файлов без извлечённого контента на этапах `small`/`large`; повторные проходы больше не пропускают такие файлы как завершённые.
- Ожидание недоступного каталога стало конечным: `catalog_wait_attempts` × `catalog_wait_seconds`, затем `RuntimeError`.
- `IndexStateDB` получил таблицу `index_config`; индексатор сохраняет и валидирует `embedding_model`, `vector_size`, `collection_name`, при несовпадении требует `--recreate`.
- `scope_files` сортируется по `mtime DESC`, чтобы новые и изменённые файлы проходили первыми.
- Добавлен конфиг `index_exclude_patterns` и фильтрация при основном проходе и cleanup.
- Добавлены экстракторы `.txt` и `.csv`, расширения включены в `SUPPORTED_EXTENSIONS`.
- Добавлен конфиг `ocr_max_image_pages`; `MAX_IMAGE_PAGES` остался только дефолтом для обратной совместимости.
- `process_file()` больше не использует legacy `_index_metadata`/`_index_content`: одиночная индексация пишет metadata+content одним batch-upsert.
- `IndexStageRunner` теперь обращается к `RAGIndexer` явно через локальный `indexer`, без магического proxy.
- Runtime-состояние файлов осталось только в `IndexStateDB`; тесты переведены на fake `state_db`.
- Размер provenance-группы чанков задаётся через `chunk_group_size` с дефолтом `4`.
- Добавлены pure-Python extractor'ы для `.rtf` и `.pptx`, а `.doc` читается best-effort через `antiword` или LibreOffice.
- `.zip` раскрывается в pipeline как набор отдельных entries; state key хранится как `archive.zip::member`, а payload path как `archive.zip/member`.
- `IndexStateDB` получил таблицу `failed_paths`; ошибки чтения сохраняются со stage `error`, retry-счётчиком и `next_retry_at`.
- `--dry-run` строит план `new` / `changed` / `stage_upgrade` / `retry_error` без batch-encode, Qdrant upsert и state mutations.
